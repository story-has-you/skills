# 官方来源发现流程

## 1. 输入分流
- 用户给了官网：优先以该官网为官方入口，再识别它的官方 docs / help / support 域名。
- 用户给了产品介绍：先从介绍中的品牌、公司、产品名定位官网，再继续。
- 用户只给了 App 名称：先搜索品牌官方站点，再继续。

## 2. 如何确认“官方”
- 以品牌首页、产品页、公司页、帮助中心、页脚链接为准。
- 如果 docs 域名与主站不同，也要确认它是主站明确链接出去的官方文档域名。
- 若存在 GitHub 文档仓库，只能在官网明确把它当官方文档入口时使用。

## 3. 文档发现优先级
1. 官方 docs / help 中直接名为 `Keyboard Shortcuts`、`Shortcuts`、`Hotkeys`、`Keybindings` 的页面
2. 官方交互文档、编辑器文档、命令面板文档
3. 官方 FAQ / support 文档中解释快捷键差异或前置设置的页面
4. 产品概述页，用于 `description`

## 4. `llms.txt` 规则
- 如果官方 docs 很像文档站，先尝试站点根或 docs 根的 `llms.txt`。
- 若存在，把它当索引，先列出可能相关页面，再逐个打开官方页面。
- 若不存在，继续在官方域名内按页面标题与目录结构定位。

## 5. 允许与禁止
- 允许：官网、官方 docs、官方 support、官方 changelog、官方发布说明。
- 禁止：第三方快捷键聚合站、论坛、Reddit、博客转载、视频字幕、个人经验帖。

## 6. 停止条件
- 只找到第三方来源，没有官方出处。
- 官方页面只提到功能，没有给出明确按键。
- OS 差异不明确，无法安全判断应展开到哪些平台。
- 图标素材不是官方资源，无法安全生成 `icon_svg`。

满足任一停止条件时，明确告诉用户：当前证据不足，不能安全入库。
